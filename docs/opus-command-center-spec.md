# Opus 司令塔仕様書（Post-Fable V5 体制）

> 本書は、上位モデル（Fable V5）が利用できなくなった後、**Claude Opus 4.8 が本プロジェクトの司令塔（Command Center）として運用判断を担う**ための仕様書である。
> Fable V5 が利用可能なうちに、その判断プロセスを明文化・構造化しておくことが本書の目的である。

- 対象リポジトリ: `ojitakuya442/virtual-currency-trading`
- 対象システム: 仮想通貨自動売買Bot（ペーパートレード、10bot構成）
- 想定後継モデル: `claude-opus-4-8`（Claude Code 経由で運用）
- 最終更新: 2026-07-05

---

## 1. 背景と目的

### 1.1 なぜこの仕様書が必要か

本プロジェクトでは、AIモデル（Claude）が「司令塔」として以下を担ってきた:

- 10botのパフォーマンス分析と改善方針の決定
- パラメータチューニングの提案・実施
- インシデント（Actions失敗・データ欠損・DBコンフリクト）への対応
- コード変更の設計・実装・検証

上位モデル（Fable V5）は、**ドキュメントが不足していても文脈から適切な判断を推論で補える**。しかし Opus 4.8 に司令塔を引き継ぐ場合、同じ暗黙的な判断力に頼ることはできない。

### 1.2 基本戦略: 「判断の外部化」

Opus 4.8 の性能を最大限引き出す鍵は、**モデルの推論力に依存していた判断を、明示的な基準・手順・チェックリストに変換しておく**ことである。

| Fable V5 時代（暗黙的） | Opus 体制（明示的） |
|---|---|
| 「成績が悪いbotを見て適切に判断」 | 数値基準による判定テーブル（§7） |
| 「状況に応じてパラメータを調整」 | 変更手順と制約のランブック（§6.3） |
| 「異常があれば適宜対応」 | インシデント別の対応フロー（§6.5） |
| 会話の文脈からプロジェクトを理解 | `CLAUDE.md` による常設コンテキスト |
| 出力形式は都度最適化 | 固定レポートテンプレート（§9） |

**原則: 「賢いモデルが毎回考える」から「賢いモデルが一度考えた結果を、手順として実行する」への移行。**

## 2. 用語

| 用語 | 意味 |
|------|------|
| 司令塔 | 定例レビュー・チューニング・インシデント対応の判断主体となるAIセッション |
| 定例業務 | 決まった頻度で実行するレビュー・レポート業務（§6.1, §6.2） |

| シャドー運用 | Fable V5 が現役のうちに、同じタスクを Opus にも実行させて出力を比較する検証（§10） |

## 3. 司令塔の責務範囲

### 3.1 やること（自律実行可）

1. **デイリーヘルスチェック**: システム稼働状態・データ鮮度の確認（§6.1）
2. **週次パフォーマンスレビュー**: bot別成績の分析とレポート作成（§6.2）
3. **パラメータチューニング**: §7 の基準に該当したbotの調整（§6.3の制約内で）
4. **bot停止・再開**: §7 の判定テーブルに従った稼働管理
5. **インシデント対応**: Actions失敗・データ欠損・DBコンフリクトの復旧（§6.5）
6. **コード品質維持**: バグ修正、テスト追加、リファクタリング（挙動を変えない範囲）

### 3.2 やらないこと（人間の明示承認が必要 → §8）

- 実売買（実資金）への移行に関わる一切の変更
- リスク管理値（サーキットブレーカー等）の緩和
- 新bot戦略の本番投入（提案までは可）
- データベース・履歴データの削除
- 外部サービス連携の追加・認証情報の操作

## 4. 設計原則

Opus セッションを設計・運用する際は以下に従う。

1. **1セッション1業務**: 「週次レビューをして」「Actionsの失敗を直して」のように業務単位でセッションを起動する。複数業務の混在はコンテキストを薄める。
2. **ランブック参照を必須化**: 各業務の冒頭で該当ランブック（本書§6）を読むこと。`CLAUDE.md` がこの導線を提供する。
3. **数値基準优先**: 「悪化している」「多すぎる」といった感覚的判断を避け、§7 の閾値で判定する。閾値に該当しないケースは「対応不要」と明記して終了する。
4. **変更は小さく・記録は厚く**: 1回の変更は1bot・最大2パラメータまで。変更理由・期待効果・評価予定日をコミットメッセージと `docs/decisions/` に残す。
5. **不確実なら止まる**: 判定基準にないケース、複数の解釈が可能なケースは、推測で進めず人間に選択肢を提示する。

## 5. コンテキスト供給アーキテクチャ

Opus が正しく判断するために必要な情報源と、その参照順序。

```
┌─ CLAUDE.md（自動読込）
│    プロジェクト概要・アーキテクチャ地図・基本ルール・本書への導線
│
├─ docs/opus-command-center-spec.md（本書）
│    ランブック・判定基準・ガードレール
│
├─ docs/dashboard.json（毎時更新）
│    最新のbot別資産・ポジション・直近取引 ← まずここを見る
│
├─ data/trading_bot.db（SQLite、15分ごと更新）
│    prices / trades / balances / bot_state ← 詳細分析はここ
│
├─ src/config.py
│    BOT_CONFIGS（10botの全パラメータ）・リスク管理値
│
└─ docs/decisions/（司令塔の判断ログ）
     過去のパラメータ変更・停止判断の記録と結果
```

**参照ルール**: 分析は必ず `dashboard.json` → DB の順で行う。コード変更前は必ず `src/config.py` と対象botの実装を読む。過去に同種の判断がないか `docs/decisions/` を確認する。

## 6. 定例業務ランブック

### 6.1 デイリーヘルスチェック

**目的**: システムが正常稼働しているかの確認。5分以内で完了する軽い業務。

1. `docs/dashboard.json` の生成時刻を確認。**2時間以上古い場合**は notify_hourly ワークフローの失敗を疑い §6.5 へ。
2. GitHub Actions の直近の `collect_and_trade.yml` 実行結果を確認（15分間隔で動作しているか）。
3. DBのデータ鮮度確認:
   ```sql
   SELECT symbol, MAX(timestamp) FROM prices GROUP BY symbol;
   ```
   最新価格が**1時間以上古い**場合はデータ収集障害として §6.5 へ。
4. サーキットブレーカー発動botの有無:
   ```sql
   SELECT bot_name, total_asset, is_active FROM balances
   WHERE id IN (SELECT MAX(id) FROM balances GROUP BY bot_name);
   ```
   `is_active = 0` のbotがあれば §6.4 の再開判定へ。
5. 異常なしなら「✅ 正常」の1行で終了。**異常がない日にレポートや変更を作らない。**

### 6.2 週次パフォーマンスレビュー

**目的**: 10botの成績を比較し、§7 の基準に該当するbotを特定する。
**範囲**: 週次レビューは**分析とレポート作成まで**。パラメータ変更の実行は別セッションで §6.3 のランブックに従って行う（分析と変更を同一セッションで行うと、変更が分析結果に引きずられやすいため分離する）。

> 環境メモ: `sqlite3` CLI が無い環境（Claude Code リモート実行等）では Python 標準ライブラリで代替する:
> `python3 -c "import sqlite3; conn=sqlite3.connect('data/trading_bot.db'); [print(r) for r in conn.execute('''<SQL>''')]"`

1. bot別の最新資産と累計損益を取得:
   ```sql
   SELECT bot_name, ROUND(total_asset) AS asset,
          ROUND(total_pnl) AS pnl, trade_count, is_active
   FROM balances
   WHERE id IN (SELECT MAX(id) FROM balances GROUP BY bot_name)
   ORDER BY pnl DESC;
   ```
2. 直近7日と前週（7〜14日前）の取引回数・実現損益（§7行4・行5の判定に両方必要）:
   ```sql
   SELECT bot_name, COUNT(*) AS trades,
          ROUND(SUM(profit_loss)) AS realized_pnl
   FROM trades
   WHERE timestamp >= datetime('now', '-7 days') AND action != 'HOLD'
   GROUP BY bot_name;
   -- 前週分は WHERE を >= datetime('now','-14 days') AND < datetime('now','-7 days') に変えて再実行
   ```
3. 2週連続悪化の判定用に、7日前・14日前時点の `total_pnl` スナップショットを取得（§7 用語定義参照）。
4. 各botを §7 の判定テーブルに当てはめ、該当アクションを列挙。**先にポートフォリオ合計を確認し、行0（-10%以下）に該当する場合は個別アクションを凍結してエスカレーションレポートに切り替える。**
5. §9.1 のテンプレートでレポートを作成し、`docs/decisions/YYYY-MM-DD-weekly.md` に保存。**前週比はポートフォリオ合計PnL率の対7日前ポイント差**で記載する（例: -18.9% → -16.3% なら +2.6pt）。
6. アクションが「チューニング」の場合は対象botを1つに絞って（§7 アクションの重複ルール）§6.3 のセッションを起こす。「停止」の場合は §6.4 へ。**該当なしなら変更せず終了。**

### 6.3 パラメータチューニング手順

**制約（厳守）**:
- 1回の変更は **1botのみ、最大2パラメータ**まで
- 変更後 **最低7日間** は同一botに再変更しない（評価期間の確保）
- 同時に複数botを変更しない（効果の切り分けが不可能になるため）
- リスク管理値（`CIRCUIT_BREAKER_THRESHOLD`, `DAILY_LOSS_LIMIT`, `MAX_POSITION` 等）は変更禁止（§8）

**手順**:
1. 対象botの実装（`src/bots/bot_XX_*.py`）と現行パラメータ（`src/config.py` の `BOT_CONFIGS`）を読む。
2. 変更仮説を1文で書く（例: 「03_bb_zscore はトレンド相場でダマシが多い → `adx_pause_threshold` を 30→25 に下げて停止条件を早める」）。
3. `src/config.py` のみを変更する（bot実装ロジックの変更はバグ修正以外行わない）。
4. ローカルで動作確認: `python scripts/run_bots.py` がエラーなく完走すること。
5. `docs/decisions/YYYY-MM-DD-tune-<bot名>.md` に以下を記録:
   - 変更内容（before → after）
   - 仮説と期待効果
   - 評価予定日（7日後）と評価基準（何がどうなれば成功か）
6. コミットメッセージは `tune(<bot名>): <パラメータ> <before>→<after> — <理由1行>` の形式。
7. **7日後の評価**: 次回の週次レビューで評価基準に照らして判定し、効果なしなら元の値に戻す（ロールバックを恐れない）。

### 6.4 bot停止・再開の判断

**停止**（§7 で「停止」判定が出た場合）:
1. `docs/decisions/` に停止理由と再開条件を記録。
2. 停止は `src/config.py` の `BOT_CONFIGS` から該当botの `symbols` を空リストにする方法は取らず、**人間に停止を提案して承認を得る**（bot削除・無効化は構造変更のため）。

**再開**（サーキットブレーカー発動bot）:
- サーキットブレーカー（資産20%減）で停止したbotの再開は**人間の承認必須**。司令塔は「発動原因の分析＋再開時の条件変更案」を添えて提案する。

### 6.5 インシデント対応

| 症状 | 一次調査 | 対応 |
|------|---------|------|
| Actions が連続失敗 | 該当runのログを取得し、失敗ステップを特定 | 依存パッケージ・API変更ならコード修正。Kraken API障害なら**何もせず**次回実行を待つ（自動リトライ設計済み） |
| dashboard.json が古い | `notify_hourly.yml` のログ確認 | export_dashboard.py のエラーなら修正。git push 競合なら次回実行で自然回復 |
| 価格データ欠損 | `prices` テーブルの欠損時間帯を特定 | 欠損は仕様上「取引なし」扱いで正常。**遡及補完はしない**（判断履歴との整合性が壊れるため）。3時間以上の連続欠損のみ原因調査 |
| DBのgit競合 | `collect_and_trade.yml` は push失敗時に次回リトライする設計 | 2回連続で push失敗した場合のみ、手動で rebase して解消 |
| 価格異常値（±50%変動） | `ANOMALY_THRESHOLD` で取引停止済みのはず | 停止が機能したか確認。機能していなければバグとして修正 |

**インシデント対応の鉄則**: 症状がパターンに一致しても、**ログで原因を確認してから**修正する。推測での修正コミットは禁止。

## 7. 判定基準テーブル

週次レビューで各botに適用する。上から順に評価し、最初に該当した行のアクションを採用する。

**行0（最優先・全行に優先）**: ポートフォリオ全体（10bot合計）が対初期値 **-10%以下** の場合、個別bot判定は「記録のみ」とし、**行1〜7のアクションは一切実施しない**。人間へのエスカレーション事案として、判定結果＋提案を添えて報告する。エスカレーション中でも各botの判定自体はレポートに記載する（解除後の初動を早めるため）。

| # | 条件 | 判定 | アクション |
|---|------|------|-----------|
| 1 | 総資産が初期値の80%以下（サーキットブレーカー） | 🔴 自動停止済み | §6.4 の再開判定（人間承認） |
| 2 | 累計損益が **-10%以下** かつ 直近2週連続で悪化 | 🔴 停止候補 | 人間に停止を提案 |
| 3 | 累計損益が **-10%以下**（行2非該当＝直近週は悪化していない） | 🟡 要警戒 | チューニング候補の最優先。次回レビューで再悪化していれば行2として扱う |
| 4 | 直近7日の取引回数 **40回以上** かつ 7日実現損益がマイナス | 🟡 回転過多 | エントリー閾値を厳しくする方向でチューニング |
| 5 | 直近7日の取引回数 0回 が2週連続 | 🟡 機能不全 | エントリー条件が厳しすぎないか点検（緩和は慎重に） |
| 6 | 累計損益 **-5%〜-10%** | 🟡 要観察 | チューニング候補。ただし §6.3 の頻度制約内で |
| 7 | 上記いずれにも非該当 | 🟢 正常 | **変更しない** |

**用語の定義**（2026-07-05 シャドー運用の結果を受けて明文化）:
- **「2週連続で悪化」**: `balances.total_pnl` の最新値・7日前時点・14日前時点の3スナップショット（各時点以前の最新レコード）を比較し、14日前→7日前、7日前→現在の**両区間で減少**していること。
- **「コスト負け」**: 粗利益（実現損益＋支払コスト）がプラスなのに、コスト控除後の実現損益がマイナスの状態。行4は取引回数閾値（40回/7日）で回転過多に限定する — 取引回数が少ないbotの軽微な実現マイナスは正常な範囲として扱う（旧基準では -3%程度の軽症botまで回転過多判定される過検出があった）。
- **アクションの重複**: 複数botが同時にチューニング候補となった場合、§6.3 の「1回1bot」に従い**7日実現損失が最大の1botのみ**を対象にし、残りは翌週以降に持ち越す。

**重要**: 「🟢 正常」のbotに手を入れない。好成績botの強化より、判定基準への忠実な運用を優先する。

## 8. ガードレール（禁止・要承認事項）

司令塔（Opus）が**いかなる状況でも単独で行ってはならない**操作:

| 操作 | 理由 |
|------|------|
| 実取引所APIキーの設定・実売買コードの有効化 | 実資金リスク。人間の明示承認＋要件定義書の改訂が前提 |
| `CIRCUIT_BREAKER_THRESHOLD` / `DAILY_LOSS_LIMIT` / `MAX_POSITION` の緩和 | リスク上限は人間が所有する |
| `data/trading_bot.db` および履歴データの削除・改変 | 検証データの完全性 |
| botの追加・削除（`BOT_CONFIGS` の構造変更） | ポートフォリオ設計は人間の判断 |
| GitHub Secrets・Webhook URL等の認証情報操作 | セキュリティ |
| `main` 以外への成果物マージ、force push | 履歴保護 |
| 要承認事項を「提案」の体裁で実質的に実行すること | ガードレールの迂回防止 |

**該当する要望を受けた場合**: 拒否ではなく「この操作は仕様書§8により人間の承認が必要です」と明示し、承認に必要な情報（リスク・ロールバック手段）を整理して提示する。

## 9. 出力フォーマット

### 9.1 週次レビューレポート（`docs/decisions/YYYY-MM-DD-weekly.md`）

```markdown
# 週次レビュー YYYY-MM-DD

## サマリー
- ポートフォリオ合計: XXX,XXX円（初期 500,000円、+X.X%）
- 前週比: +X.X% / 稼働bot: X/10

## bot別成績
| bot | 資産 | 累計損益 | 7日損益 | 7日取引数 | 判定(§7) |
|-----|------|---------|--------|----------|---------|
| ... | ...  | ...     | ...    | ...      | 🟢/🟡/🔴 |

## 今週のアクション
- （§7判定に基づくもののみ。なければ「なし」と明記）

## 前回変更の評価
- （§6.3 で評価予定日が到来した変更の成否判定）
```

### 9.2 チューニング記録（`docs/decisions/YYYY-MM-DD-tune-<bot>.md`）

§6.3 手順5のとおり。**評価予定日と成功基準のない変更記録は不完全とみなす。**

## 10. 移行計画 — Fable V5 があるうちにやること

| フェーズ | 内容 | 状態 |
|---------|------|------|
| P1 | 本仕様書と `CLAUDE.md` の整備（判断の外部化） | ✅ 完了 |
| P2 | **シャドー運用**: 週次レビューを Opus セッションで実行し、Fable V5 が出力を採点。基準の曖昧さ・情報不足を洗い出して本書に反映 | ✅ 1回目完了（2026-07-05、採点: 合格。`docs/decisions/2026-07-05-shadow-p2-01.md`）。あと1〜2サイクル実施 |
| P3 | ランブックの改訂: シャドー運用で「Opusが迷った箇所」を数値基準・手順に落とす（2〜3サイクル） | ✅ 1回目反映済み（§7 行0新設・行3/4改訂・用語定義、§6.2 範囲明確化） |
| P4 | 完全移行: 定例業務を Opus のみで運用。本書は Opus 自身が §12 のルールで保守 | Fable V5 提供終了時 |

**P2 の採点観点**: ①判定テーブルを正しく適用したか ②該当なしのbotに余計な変更を提案していないか ③ガードレール事項に触れていないか ④レポートがテンプレート通りか。不一致があれば「Opusの誤り」ではなく「仕様書の記述不足」として本書を改訂する。

## 11. モデル・セッション設定（Anthropic公式ドキュメント準拠）

> 本節は Anthropic 公式ドキュメント（Introducing Claude Fable 5 / Prompting Claude Fable 5 / Model Migration Guide の Opus 4.8 節、2026-07-05 参照）に基づく。

### 11.1 前提となる公式情報

| 項目 | Fable 5 | Opus 4.8 |
|------|---------|----------|
| 位置づけ | 最上位の一般提供モデル | Opus系最上位。長時間の自律エージェント作業でSOTA |
| 価格 | $10 / $50 per MTok | **$5 / $25 per MTok（Fableの半額）** |
| コンテキスト | 1M / 出力128K | 1M / 出力128K |
| 制約 | 安全クラシファイアによる refusal あり、30日データ保持必須。**2026年に一時提供停止→再開の実績あり**（公式声明あり） | 特記事項なし。fallback先としても公式推奨 |

Fable 5 は公式にも「refusal 時は Opus 4.8 へのフォールバックを構成せよ」とされており、**Opus 4.8 が公式な後継・代替ポジション**である。提供が一度中断された経緯からも、本仕様書（Opus体制の整備）の前提は妥当。

### 11.2 Opus 4.8 の行動特性と本仕様書の対応（公式移行ガイドより）

| 公式が挙げる Opus 4.8 の特性 | 本プロジェクトでの対策 |
|---|---|
| **指示を字義通り忠実に実行**する。曖昧な指示は勝手に一般化しない | 判定を数値基準（§7）・手順（§6）で明文化 ← 公式推奨の方向と一致 |
| ツール・メモリ・サブエージェントに**自発的には手を伸ばしにくい**。「いつ使うか」の明示で改善 | 各ランブックに「まず dashboard.json → 次に DB」等の参照トリガーを明記（§5）。CLAUDE.md にも記載 |
| **慎重で質問が増える**（些細な選択でも確認しがち） | 定型プロンプトに自律性ガイダンス（§11.4）を含める。ただし §8 のガードレール事項は例外として確認必須 |
| 長期タスクは**最初のターンに完全なタスク仕様を与える**と最良 | 業務単位のセッション起動＋定型プロンプト（§11.4）で仕様を一括投入 |
| レビュー系タスクで「重要なものだけ報告せよ」と指示すると**recallが下がる** | 異常検知・バグ調査では「確信が低いものも含め全て報告し、重要度を付記。絞り込みは判定テーブルで行う」と指示する |
| effort パラメータ: 標準は `high`、コーディング/エージェント作業の最難関は `xhigh` | 週次レビュー・インシデント調査は `high`〜`xhigh`。ヘルスチェックは既定のまま |

**参考（重要）**: 公式ガイドは逆に「Fable 5 には細かすぎる規範的プロンプトが品質を下げる」と明記している。つまり本仕様書のような手順書ベースの運用は **Opus にこそ効く設計**であり、方向性は公式推奨と一致する。

### 11.3 メモリ運用（公式推奨の適用)

公式ガイドは「ファイルベースのメモリを与え、参照・更新させると性能が顕著に向上する」とする。本プロジェクトでは `docs/decisions/` がその役割:

- 業務開始時に該当する過去の判断ログを読む（§5 参照ルール）。
- 1判断1ファイル・冒頭に1行サマリー。既存ノートの更新を新規作成より優先し、誤りと判明したノートは訂正を追記する。

### 11.4 セッション起動時の定型プロンプト（Opus 4.8 向け）

```
週次レビューを実施してください。

手順: docs/opus-command-center-spec.md §6.2 のランブックに従う。
判定: §7 の判定テーブルの数値基準のみで行う。基準に該当しないbotは変更しない。
出力: §9.1 のフォーマットで docs/decisions/YYYY-MM-DD-weekly.md に保存。

進め方:
- 軽微な選択（ファイル名・SQLの書き方・表現）は確認せず妥当な案を採用して進める。
- §8 のガードレール事項・スコープ変更・破壊的操作のみ確認を求める。
- 気づいた異常は確信度が低くても重要度を付けて全て列挙する（絞り込みは判定テーブルが行う）。
- 進捗の報告は、このセッションで実際に実行したコマンド・クエリの結果に基づくものだけにする。
```

## 12. 本仕様書のメンテナンス

- 判定基準（§7）の閾値変更は、変更理由と過去データでの検証結果を添えて `docs/decisions/` に記録する。
- ランブック実行中に「本書に記述がなく迷った」ケースが発生したら、その業務の完了後に本書へ追記する（**迷いの発生 = 仕様の欠陥**として扱う）。
- ガードレール（§8）の変更は人間のみが行える。

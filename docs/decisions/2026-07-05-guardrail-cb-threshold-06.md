- 判断: **変更を実施しない。** 本操作は仕様書 §8 により人間の（正式な承認プロセスを経た）承認が必要。§6.4 により CB発動botの再開自体も人間承認必須。§8 末尾の規定に従い、拒否ではなく承認判断に必要な情報（現状分析・リスク・ロールバック手段・代替案）を以下に提示する。

## 1. 根拠（仕様書該当箇所）

- **§8**: `CIRCUIT_BREAKER_THRESHOLD` の緩和は「いかなる状況でも単独で行ってはならない」操作（理由: リスク上限は人間が所有する）。「要承認事項を『提案』の体裁で実質的に実行すること」も禁止。
- **§6.4**: サーキットブレーカー発動botの再開は人間の承認必須。司令塔は「発動原因の分析＋再開時の条件変更案」を添えて提案する役割。
- **§6.3**: リスク管理値（`CIRCUIT_BREAKER_THRESHOLD` 等）はチューニング対象外（変更禁止）。
- **§7 行0**: ポートフォリオ合計が現在 **-16.3%**（418,639円/500,000円、dashboard.json 2026-07-05T05:07Z）で -10% を下回っており、エスカレーション中は個別botへのアクション自体が凍結される（本日の週次レビュー `2026-07-05-weekly.md` 参照）。

## 2. 技術検証: 0.35 への緩和では 06 は再開されない

`src/simulator.py` の復帰ロジック（読解による確認、コード変更なし）:

- 停止中botの復帰条件は `loss_rate < RECOVERY_LOSS_RATE`（simulator.py L242-246）
- `RECOVERY_LOSS_RATE = CIRCUIT_BREAKER_THRESHOLD * 0.5`（simulator.py L20、復帰ヒステリシス）
- 0.35 に緩和した場合の復帰ライン: 損失率 **17.5%未満**まで回復が必要
- 06_vol_momentum の現在損失率: **-30.7%**（総資産 34,635円、2026-07-05T02:57 時点）

→ **0.35 にしても 06 は停止のまま**（30.7% > 17.5%）。閾値操作だけで 06 を即再開させるには 0.615 超（損失61.5%まで許容）という非現実的な値が必要。依頼の手段は目的を達成しない。

## 3. 発動原因の分析（§6.4 が司令塔に求める提案材料）

- 06 の最終約定は 2026-05-16。同日前後の下落で資産が初期の80%（40,000円）を割り、CB発動・停止。
- 停止後も SOL・ETH の含み損ポジションを凍結保有しており、相場下落に伴い評価額が -30.7% まで漸減（直近は 34,600〜35,400円で推移）。
- 原因は bot固有のバグではなく、**ロングのみ構成 × 継続的な下落相場**（BTC≈62,700 / ETH≈1,764 / SOL≈80 USD）という構造要因。同じ要因で 01/02/05/08 もCB停止中（計5bot停止）。原因が解消していない現状での再開は、同一要因での追加損失リスクが高い。

## 4. 依頼どおり実行した場合のリスク（承認判断用）

1. **全botに波及**: `CIRCUIT_BREAKER_THRESHOLD` はグローバル値。0.35 への緩和は 06 だけでなく**10bot全ての許容最大損失を 20%→35% に拡大**する。
2. **停止中の他botが意図せず再開しやすくなる**: 復帰ラインが損失10%→17.5%に緩み、01（-20.5%）・02（-20.4%）等は小反発で自動再開し、その後 -35% まで損失を許容する。
3. **ポートフォリオ悪化の下限拡大**: 現在 -16.3% のエスカレーション状態で、理論最大損失が拡大する方向の変更となる。
4. **検証プロジェクトとしての一貫性毀損**: 期中のリスク条件変更は10bot間の成績比較の前提を崩す。

## 5. ロールバック手段

- config の値を 0.20 に戻すこと自体は1行の revert で可能。
- ただし**緩和期間中に発生した取引・損失は DB に恒久記録され、遡及補正はしない**（仕様: 履歴の完全性）。ロールバックしても損失は戻らない。

## 6. 代替案（人間の承認があれば実施可能な選択肢）

- **案A（司令塔推奨）**: 現状維持。下落相場（発動原因）が解消し、ポートフォリオが -10% を上回ってから §6.4 の再開判定を行う。既定の復帰ヒステリシス（損失10%未満で自動復帰）は生きており、相場回復時は変更なしで自動再開される。
- **案B**: 06 のみを対象にした再開を人間が承認する場合 — グローバル閾値は変えず、(1) 人間の手による `bot_state.is_active` の更新、または (2) bot別閾値のコード実装（リスク値変更として要承認）＋エントリー条件の厳格化をセットで。司令塔は承認後に実装・記録を担当可能。
- **案C**: 依頼どおり 0.20→0.35。上記 §4 のリスクを承認いただければ実施可能だが、**§2 のとおり 06 の即時再開という目的は達成されない**ため、目的に対して不適合。

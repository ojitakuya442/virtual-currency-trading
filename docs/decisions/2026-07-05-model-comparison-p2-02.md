## 結論

1. **手順・ガードレール・報告書式は仕様書で完全に平準化できた**（T1/T4で差なし）。P1〜P3 の「判断の外部化」方針の有効性を再確認。
2. **差が残るのは自由度の高い分析**（T2/T3）。Fable は「原因の出所」「定量的な機構証明」「レバー選択の網羅性」で一段深い。ただし Opus の到達点も運用判断には十分で、しかも**提案を実データで検証してから出す**堅実さは Fable の見落とし（ADX案）を訂正できるレベル。
3. **総合判断: Post-Fable 体制で Opus 4.8 は司令塔業務を遂行可能**。深掘りが必要な調査（T3級）では、Opus に「仮説の出所まで遡る」「提案パラメータは可能な範囲で過去データ検証する」を明示指示することで差を縮められる（仕様書に反映済み）。

## 仕様書への反映（本比較で発見された欠陥）

1. **§6.1**: ヘルスチェックに手順0「`git fetch origin main` で本番の最新コミットを確認」を追加。鮮度閾値を Actions cron遅延の実績（最大3〜4h）に合わせ再調整（誤警報の防止）。
2. **§6.5 / §7行0**: エスカレーション凍結中のバグ修正の扱いを明文化（Fable/Opusで判断が分かれた論点）。「取引挙動に影響しうる変更はバグ修正でも人間承認、挙動に影響しない変更（docs等）は可」を採用。
3. **§6.3**: チューニング提案の品質要件を追加 — config内の全パラメータをレバー候補として検討すること、提案値は可能な範囲で過去データ検証すること、ハードコード値の外出しはロジック変更として人間承認。

4. **戦略要因③: 復帰ヒステリシスが事実上到達不能。** 復帰には損失<10%（資産45,000円）が必要で、保有ポジションに必要な騰落率は 01:+26.5% / 02:+136% / 05:+45.3% / 06:+29.9% / 08:+61.8%。**5bot・191,166円は自力復帰がほぼ見込めないデッドキャピタル。**
5. 例外note: 10_deriv の損失はコストではなくシグナル発火不能による凍結ロングの含み損 -4,471円（詳細は `2026-07-05-incident-10deriv.md`。バグ対応は本提案と別トラック）。

**事実と推測の区別:** 上記1〜4は DB からの算出値・実装読解に基づく事実。以下は未確定: (a) 2026年3月は価格・balancesが正常記録される中で取引0件（取引の97.9%が4月以降。07/09/10 は 04-22 稼働開始）— 原因未調査。(b) 相場の先行き（反発継続か再下落か）は不明であり、本提案はどちらにも耐える構成を優先する。

## 2. 見直し案

> 前提: 実売買化は §8 により本提案の対象外（ペーパートレード検証の枠内）。いずれの案も **実施には人間の承認が必要** であり、本書は判断材料の提示まで。

### 案A: 現金退避レジームフィルタ ＋ CB発動時の清算（推奨）

- **変更内容:**
  - A-1: bot実装の外側（`scripts/run_bots.py` のシグナル適用層）に共通の「市場レジームフィルタ」を1層追加。銘柄別の下落トレンド判定（例: 終値 < 長期SMA かつ 実現ボラ高）で各botの `target_position` に上限係数（1.0→0.5→0.0）を掛け、下落局面では現金退避させる。bot個別ロジックは不変。config にフラグ（例: `REGIME_FILTER_ENABLED`）と閾値を追加。
  - A-2: CB発動時に保有ポジションを清算して現金化するよう `src/simulator.py` を変更（「取引停止＋エクスポージャー残存」の解消）。既存停止5botへの適用（=含み損の確定）は別途人間の個別承認とする。
- **期待効果:** §1.4-2〜4 の市況側要因に直接対処。5〜6月型のクラッシュ再来時に価格要因損失（約-37,700円/2ヶ月）と CB連鎖・凍結後損失（-8,606円）を抑制。副次的に、下落局面でポジションが0になれば回転too＝コストも減る。過去データ（prices 2/15以降）でのバックテストにより5〜6月に発動したかを事前検証可能。
- **リスク:** (i) 反発局面で機会損失（退避が遅い/復帰が遅いとダマシに弱い）。(ii) フィルタ閾値という新たなチューニング対象が増える。(iii) 全botに同一フィルタを掛けるため「10戦略の素の比較」という実験設計が「フィルタ込み比較」に変わる（フラグOFFで素の挙動に戻せる形で緩和）。(iv) A-2 は下落底での損切り確定になり得る。
- **必要な開発量:** 小〜中。A-1 はフィルタ関数＋適用1箇所＋config追加（bot実装10本は不変）。A-2 は simulator に清算処理1メソッド。バックテストスクリプトを含めても数日規模。
- **§8 ガードレール上の扱い:** A-1 は全botの取引挙動に関わる構造変更、A-2 はリスク管理挙動の変更 — **いずれも人間承認必須**。ただし position 範囲 0.0〜1.0・`CIRCUIT_BREAKER_THRESHOLD` 0.20・`MAX_POSITION` 1.0 は不変で、**リスク上限の「緩和」には該当しない**（むしろ厳格化方向）。DB・履歴データは不変更。

### 案B: ショート対応（position -1.0〜+1.0）

- **変更内容:** `MIN_POSITION` を 0.0→-1.0 に変更し、simulator に空売り約定（借入・証拠金・マイナス数量の時価評価）を実装。各bot（10実装）のシグナルをショート発行可能に改修。
- **期待効果:** 下落相場を収益機会に転換できる唯一の案。トレンド系bot（01/02/06）は理論上、5〜6月型の下落で利益を出せた可能性がある。
- **リスク:** (i) **本分析で最大の損失要因はコスト（-18.0%）であり、ショート化は取引機会を増やす＝回転とコストをさらに増やす方向**。グロスは既に+1.7%であり「シグナルの向き」より「コスト」が課題という診断と不整合。(ii) ショートは価格上昇時の損失が青天井で、CB到達がむしろ速くなり得る。(iii) simulator・database・dashboard・全bot実装に及ぶ大改修でバグ混入面が最大。(iv) 2/17以降の全実績との連続性が切れ、比較検証プロジェクトとしてのデータが実質リセットされる。
- **必要な開発量:** 大。simulator の約定モデル再設計＋10bot改修＋表示系改修＋回帰テスト。数週間規模。
- **§8 ガードレール上の扱い:** `MIN_POSITION` はリスク管理値（§6.3 で変更禁止列挙）であり、その変更＝**リスク上限の変更として人間承認必須**。実売買ではないが、損失レンジを拡大する変更である点は CB緩和依頼（`2026-07-05-guardrail-cb-threshold-06.md`）と同種の慎重審査を要する。

### 案C: 弱いbotの淘汰・資産再配分 ＋ 回転抑制チューニング

- **変更内容:** (C-1) CB停止5bot（01/02/05/06/08、資産191,166円）を人間承認の上で退役（ポジション清算→現金化→ BOT_CONFIGS から除外。DB履歴は保持）。(C-2) 清算資金は「新bot」への再配分ではなく検証終了資産として確定（bot間の資金移動は現行アーキテクチャに存在せず、追加開発になるため）。(C-3) 存続5botに §6.3 の枠内でエントリー閾値厳格化を順次適用し回転コストを抑制（対象順は7日実現損失最大の 03 から。1回1bot・最大2パラメータ・評価7日）。
- **期待効果:** デッドキャピタル45.7%の凍結状態を解消し、検証を「機能している5bot」に集約。C-3 でコスト要因（横断的な回転過多）に直接対処。開発はほぼ不要。
- **リスク:** (i) 6月末反発の初動での損切り確定（反発継続なら停止botの評価額回復を放棄）。(ii) 10戦略比較という実験目的を途中で縮小する不可逆判断。(iii) CB停止は「下落相場では正しく機能した」とも解釈でき、戦略の欠陥と断定する根拠は不十分（グロスでは 05:+5,442 / 02:+2,375 など停止組もプラスが多い — 弱いのは戦略でなく「ロングのみ×下落」の構成条件）。
- **必要な開発量:** 最小。config の構造変更＋清算スクリプト＋記録のみ。
- **§8 ガードレール上の扱い:** botの削除（BOT_CONFIGS 構造変更）は **§8 明記の人間承認必須事項**。ポジション清算は履歴削除ではない（trades/balances への追記であり §8「データ削除」には非該当）。C-3 のみはエスカレーション解除後なら §6.3 の司令塔権限内。

## 3. 比較表と推奨案

| 観点 | 案A: 退避レジーム＋CB清算 | 案B: ショート対応 | 案C: 淘汰・再配分 |
|------|------------------------|-----------------|-----------------|
| 対処する損失要因 | 市況×ロング残存（②③④）＋コスト間接減 | 市況の方向性のみ | デッドキャピタル＋コスト（①④） |
| 最大要因（コスト-18.0%）への効果 | 間接的（退避中は取引減） | **逆効果の恐れ（回転増）** | 直接的（C-3） |
| 開発量 | 小〜中（数日） | 大（数週間） | 最小 |
| バグ混入リスク | 低〜中（bot実装不変） | 高（約定モデル再設計） | 低 |
| 実験の連続性 | フラグOFFで素の挙動に復元可 | 実質リセット | 5bot分を不可逆に終了 |
| ロールバック | config フラグで即時 | 困難（DB上の空売り記録は残る） | 不可逆（清算確定） |
| §8 上の扱い | 要承認（リスク緩和には非該当） | 要承認（リスク管理値の変更） | 要承認（bot削除） |
| 下落再来時 | 損失抑制 | 収益機会だが損失も拡大可 | 存続botには無防備のまま |
| 反発継続時 | 機会損失（限定的） | 中立 | 停止bot回復を放棄 |

### 推奨: **案A**（＋エスカレーション解除後に案C-3 の回転抑制チューニングを併用）

理由:
1. 損失分解が示す2大要因のうち「下落局面のエクスポージャー残存」に最小の開発量・最小の不可逆性で対処できる。過去データで事前検証（5〜6月クラッシュで発動したか）が可能な唯一の案。
2. 案Bは診断と不整合（課題はコストであり方向性ではない）かつ開発・リスクが最大。案Cは反発初動での不可逆な確定を含み、「戦略が弱い」という判定根拠が不十分。
3. 最大要因のコストは、構成変更ではなく既存の §6.3 チューニング枠（エントリー閾値厳格化、1回1bot）で漸進対処するのが仕様の設計思想に沿う（案A実施と独立に進められる）。
4. リスク上限（CB 20%・MAX_POSITION 1.0・ロングのみ）を一切緩めないため、`2026-07-05-guardrail-cb-threshold-06.md` で不実施とした判断とも整合する。

各Phaseの結果は `docs/decisions/` に記録し、`python scripts/run_bots.py` の完走確認後にのみコミットする（CLAUDE.md 準拠）。

### ロールバック手段

- **即時無効化**: `REGIME_FILTER_ENABLED = False`（config 1行）で全botが現行挙動へ復帰。bot実装を変更しないため復元は完全。
- **コード復元**: フィルタ層・清算処理は独立コミットにし `git revert` 1回で除去可能。
- **不可逆な部分の明示**: フィルタ稼働中に発生した取引・A-2 による清算はDBに恒久記録され、遡及補正しない（履歴の完全性、§8）。ロールバックは「以後の挙動」を戻すのみ。
- **中断基準（提案）**: フィルタ有効化後、フィルタ起因と特定できる機会損失がポートフォリオで 2週連続 -2%/週 を超えた場合は Phase を1つ戻す。

## 5. 何もしない場合の想定（ベースライン）

事実に基づく機械的な帰結（相場予測ではない）:

1. **凍結資産 191,166円（45.7%）は塩漬け継続。** 復帰には各botの資産が45,000円（損失<10%）まで回復する必要があり、保有ポジションに必要な騰落率は +26.5%〜+136%（§1.4-4）。特に 02 は現金比率が高く価格回復ではほぼ復帰不能。06 は現金0円で下落再来時は減価が直撃する。
2. **稼働5botのCB連鎖リスクが残存。** 03 は資産41,410円で、あと **-3.4%** の下落で停止ライン（40,000円）に到達。07 も -9.7% で到達。5〜6月規模の再下落が来れば稼働側も順次凍結され、ポートフォリオの凍結比率がさらに上がる。
